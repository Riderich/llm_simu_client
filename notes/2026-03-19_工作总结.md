## 技术亮点与问题解决

### 并行处理架构设计
**问题**: 大批量标注任务效率低下  
**解决方案**: 单进程多线程 + 动态并发控制
```python
with ThreadPoolExecutor(max_workers=args.workers) as executor:
    futures = {executor.submit(classify_single_case, idx, case): idx 
               for idx, case in enumerate(cases_to_process)}
```
**效果**: 4 workers 吞吐量提升约 3 倍

### 容错与断点续跑机制
**三层保障**:
1. `--resume`: 跳过已成功标注的样本（检查 `confidence > 0` 且 `rationale` 不含 `ERROR:`）
2. `--retry-failed`: 配合 `--resume`，仅重跑失败样本
3. `--max-retries N`: 单样本 API 调用失败时自动重试（指数退避）

**增量保存**:
```python
if (processed_count % args.save_every == 0) or (processed_count == len(cases_to_process)):
    save_checkpoint(data, args.output)
```

### 强制 JSON 输出 + 二次修正
**问题**: LLM 输出不合规（缺字段、格式错误）  
**解决方案**: 二次请求修正
```python
if not is_valid_json_response(response):
    # 追加 follow-up 消息强制修正
    messages.append({
        "role": "assistant",
        "content": response
    })
    messages.append({
        "role": "user",
        "content": "Your response was not valid JSON. Please output ONLY valid JSON..."
    })
    response = client.chat.completions.create(
        messages=messages,
        temperature=0  # 强制收敛
    )
```
**实战效果**: RECAP 1000 条合作样本最终 0 失败

## 数据结构设计决策

### RECAP 数据结构最终版
```json
{
  "dialogue": [
    {"role": "counselor", "content": "..."},
    {"role": "client", "content": "..."},
    {"role": "counselor", "content": "..."},
    {"role": "client", "content": "目标发言"}  // ← 包含 target_utterance
  ],
  "target_utterance": "目标发言",  // ← 单独保留便于监督训练
  "binary_label": "resistance",
  "label": "否认问题",
  "mapped_category": "B2",
  "e_subtype": "E2",  // 仅合作样本有此字段
  "no_context": false
}
```

**为什么 `dialogue` 包含 `target_utterance`？**
- Inner Monologue 生成需要"完整对话"作为输入（见 `verify_psyfire_prompt.py` 的 Prompt 设计）
- 单独保留 `target_utterance` 是为了训练时明确监督目标

**为什么移除 `context` 字段？**
- 冗余：`context = dialogue[:-1]`，可随时从 `dialogue` 切片得到
- 降低存储开销
- 简化脚本逻辑（避免维护两个字段的一致性）

## 今日踩坑与教训

### 1. conda run 在后台任务中不稳定
**现象**: 使用 `conda run -n py10 python script.py` 在 nohup 中运行时，任务被挂起  
**原因**: conda run 会在子进程中激活环境，nohup 可能无法正确处理信号  
**解决**: 直接使用 py10 的 Python 二进制路径

### 2. 增量保存频率需权衡
**初始设置**: `--save-every 10`（每 10 条保存一次）  
**问题**: 频繁写文件影响性能，且 RECAP 数据量大（5154 条）  
**优化**: 失败重跑时调整为 `--save-every 3`（因为失败样本少）

### 3. LLM 输出格式强制修正的必要性
**问题**: 即使提供了 JSON Schema，LLM 仍可能输出不合规内容  
**解决**: 实现二次请求机制 + `temperature=0` 强制收敛  
**效果**: 从 119 失败→0 失败

### 4. 文件权限异常的快速诊断
**症状**: 脚本报错 `Permission denied`  
**快速修复**: `chmod -R u+w workspace/`  
**根本原因**: 多次切换 conda run / 手动操作导致所有者不一致（未彻底排查）
