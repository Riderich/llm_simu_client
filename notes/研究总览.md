## 一、项目定位与核心目标

**研究问题**：如何训练一个能真实模拟心理咨询中来访者阻抗行为的 LLM？

**核心挑战**：现有大模型天然倾向于配合、顺从；阻抗行为需要内在动机的支撑，而非表面话术的模仿。

**整体 Pipeline**：

```
数据集构建（标注 + inner monologue 生成）
    → SFT（监督微调）
    → DPO（直接偏好优化，针对 hard case）
```

## 二、理论框架演进

### 2.1 最初方向：完整 BDI 状态机

早期参考 **Eeyore**（抑郁模拟）和 **RECAP**（阻抗检测），计划用 BDI（Belief-Desire-Intention）架构结合 3RS 理论打造动力学引擎：

- **静态 Profile (Desire)**：含 3RS 预设倾向（对抗型/撤退型）
- **影子状态 (Belief)**：对咨询师每轮干预的实时感知
- **动作空间 (Intention/Action)**：3RS 战略 + PsyFIRE 战术

**放弃原因**：动态档案追踪在长上下文中易产生"状态漂移"；强制离散决策导致 Loss 震荡；标注成本过高。

### 2.2 当前方向：内心独白推理（Inner Monologue）

**核心顿悟**：BDI 的本质是赋予模型"动机"。可以跳过显式状态分类，直接用**中间推理过程（Chain-of-Thought）**替代。

**三段式链条**：

```
阶段一：Input Context
  静态 Persona（患者背景设定）+ 咨询师最新发言

阶段二：Inner Monologue（核心）
  <internal>
  [患者在听完治疗师的话后、说出回复前的内心活动]
  </internal>

阶段三：Final Response
  基于内心活动逻辑生成的外部回复（对应 PsyFIRE 行为类别）
```

**相对 BDI 的优势**：
- 数据构建成本大幅降低（可用 Teacher Model 逆向生成内心独白）
- 完美契合 LLM 自回归本性，逻辑连贯性更强
- 避免"为了阻抗而阻抗"的机械模仿

### 2.3 Profile 设计（D-B-I-A 精简版）

参考两次会议讨论，Profile 分为**静态 + 动态**两部分：

- **静态 Profile**：背景信息、记忆，作为 system prompt 始终置于对话最上方
- **动态 Profile**：核心心理矛盾、对咨询师的信任程度、抗拒程度，随对话动态更新

## 三、相关工作简介

| 工作 | 核心思路 | 对我们的启发 |
|------|---------|------------|
| **Eeyore** | 基于 DSM-V + 贝克认知理论构建静态 Profile，两阶段 DPO（模型偏好 + 专家校准）| 训练链路可复用（OpenRLHF 框架）；DPO 惩罚"治愈幻觉" |

| **CBS（Client Behavior System）**，Hill et al. | 8 类来访者行为，实证聚为高/低参与两组 | 为 No Resistance 三分（Exploratory / Cooperative / Resolution）提供理论依据 |

## 四、标注方案

### 4.1 标注目标

针对每条 **client utterance**，标注其 **Client Behavior Space** 类别（14 类，详见 `行为空间框架.md`）。

### 4.2 标注流程

```
大模型自动标注（utterance 级别）
    → 人工全量审核与修正
    → Ground Truth Label
    → Inner Monologue 生成（按类别选择 prompt）
```

- 大模型标注相对容易，**人工全量审核是必须步骤**（F1/准确率评估依赖 label 质量）
- 参考 RECAP few-shot vs. zero-shot 标注对比结果设计 prompt（few-shot 表现更好，~88-90% 准确率）
- 目标：**两周内完成数据标注**

### 4.3 阻抗类别影响 Inner Monologue 生成策略

不同阻抗类别对应不同 prompt 模板，详见 `Inner_Monologue技术文档.md`。

## 五、训练方案

### SFT
使用构造好的 `{对话 + internal + 阻抗标签}` 数据进行监督微调，让模型学会 `先内化 → 再回复` 的生成范式。

### DPO
筛选 SFT 表现不好的 hard case，构造 Positive/Negative 对，惩罚：
- **治愈幻觉**：内心再挣扎，最终还是配合了
- **崩人设**：角色定位切换为咨询师
- **逻辑断裂**：内心独白与最终回复不一致

### GRPO（进阶探索）
如果希望模型学习**动态推理链条**（推理长度自适应），DPO 可能不够。可探索：
- 用是否存在 `<think>` 标签作为简单 reward
- 对推理链长度进行惩罚/奖励

## 六、评估体系

| 维度 | 方法 |
|------|------|
| **行为一致性** | 生成回复是否符合给定 Persona |
| **阻抗真实感** | LLM Judge 或专家，按 PsyFIRE 维度盲测 |
| **内心-言语匹配度** | 提取 `<internal>` 内容，评估是否支撑了后续回复逻辑 |
| **基线对比** | 无 inner monologue 的直接 SFT 对比；有无 role profile 的对比 |

### 待办任务（优先级排序）

**任务 A ⭐⭐⭐ 修复自动全量生成脚本的数据加载 Bug并部署**
需要修改 `verify_psyfire_prompt.py` 中由于 AnnoMI 数据结构变化而漏抓大部分真实抵抗语料的逻辑缺陷。同时需要添加过滤器，跳过像 "Hi." 这种极短无意义对话，防止模型在极简短寒暄处陷入生造心理活动的动作幻觉。修复后在服务器挂机跑全数据集的生成。

**任务 B ⭐⭐⭐ 调整标注 Pipeline 策略（迎接 RECAP 模型）**  
由于下周将获取 RECAP 官方的抵抗分类模型及其训练数据，我们当前的自动标注策略需要部分转变：
1. 优先评估 RECAP 官方模型在我们的数据集（ESConv / MESC）上的 Zero-shot 泛化能力。
2. 如果 RECAP 模型表现优异，我们将直接使用其进行阻抗侧类别抽取，大幅降低调用 GPT-4 标注的 API 成本与人工编写 LLM 标注准则的精力。
3. 对于 RECAP 模型无法覆盖的非阻抗分类（我们自研的 E1/E2/E3 类别），编写补充的 LLM 标注脚本。

**任务 C ⭐⭐ 服务器环境部署全量生成**  
代码同步服务器 → 跑通 `AnnoMI`, `ESConv`, `MESC` 三大数据集的 Inner Monologue 组装，最终形成 `{上下文 -> <internal> -> 回复 (含阻抗类别标签)}` 的 SFT 训练对齐数据。
